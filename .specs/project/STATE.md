Both app repos' `KafkaProducerFactory`/`KafkaConsumerFactory` simplified to match (SASL/IAM code removed, `AWS.MSK.Auth` package dropped) — see their own STATE.md entries. Not yet applied against real AWS or tested end-to-end; not yet PR'd (pending a live test pass, same pattern as previous infra work this session).

## 2026-07-20 session, part 3 (end-to-end deploy attempt — VPC fix, Kafka topics, real bugs found; all infra destroyed at end of session)

**MSK security group widened to VPC CIDR.** Both app repos' EC2 instances originally launched in the AWS account's *default* VPC (`module.ec2` never took a `vpc_id`/`subnet_id`) — completely separate from this platform's VPC where MSK lives, so neither instance could resolve the broker's VPC-private DNS. Fixed in both app repos (see their STATE.md) by wiring `vpc_id`/`public_subnets` (new root outputs here) via `terraform_remote_state`. That alone wasn't enough — `module.kafka`'s security group only allowed ingress from itself (broker-to-broker). Added a second ingress rule (`msk_sasl_iam_vpc`, `modules/kafka/main.tf`) allowing port 9098 from the whole VPC CIDR (`10.0.0.0/16`) — confirmed with the user before applying, since it's a real (if narrow — still VPC-internal only, never internet-facing) security widening. New `vpc_cidr` output on `modules/network`, threaded into `modules/kafka` as `var.vpc_cidr`.

**New `module.kafka_topics` — declared in code, never successfully applied against the real backend.** MSK Serverless has no topic management via AWS Console or API ("Topic management requires Amazon MSK Provisioned... Serverless clusters are currently not supported") — topics can only be created over the Kafka wire protocol itself. Added `provider "kafka"` (Mongey/kafka, `sasl_mechanism = "aws-iam"`) and a `kafka_topic` resource per topic (6 topics: `notification-requested`, its 3 retry topics, its DLQ, and `user-lifecycle-events` — names must match `RetryTopicChain.cs`/`KafkaTopics.cs` in the two app repos exactly). **This provider failed against the real cluster** in two different ways when actually applied (from inside the VPC, since the broker DNS is private): first a `Plugin did not respond` crash (missing `AWS_REGION` env var — the provider needs it explicitly even under `aws-iam` mechanism), then after fixing that, `SASL Authentication failed: Invalid authentication payload` — the provider's IAM SASL token format appears incompatible with MSK Serverless specifically (unconfirmed whether this is fixable via more config or a real upstream bug). **The topics were ultimately created via a one-off C# console tool** (self-contained publish, `Confluent.Kafka` + `AWS.MSK.Auth` — the exact same libraries/pattern already proven working in both app repos' own `KafkaProducerFactory`), run once from inside the VPC via SSH, then discarded. `module.kafka_topics` remains in this repo's tracked source as the intended long-term IaC path, but its Terraform state was never reconciled with reality (no `terraform import` was run) — **anyone re-running `terraform apply` here in the future will likely try to recreate these topics and hit the same SASL failure** until either the provider issue is root-caused or the module is replaced with something else (e.g. a null_resource + local-exec calling a small script, or accept that topic provisioning stays a manual/scripted step outside Terraform).

**Real bugs found in the app repos while chasing this** (see their own STATE.md for full detail, since the fixes live in their code): `rentifyx-communications-api` had an unconditional fail-fast requiring `AWS:Profile` even in Production (EC2 has no named profile, uses instance role) — fixed. Both repos' EC2 `data "aws_ami" "amazon_linux_2023" { most_recent = true }` caused unrelated instance replaces on every plan — fixed with `lifecycle { ignore_changes = [ami] }`. Both repos' CORS temporarily set to accept any origin (`SetIsOriginAllowed(_ => true)`) — no real frontend exists yet, previous config only allowed `http://localhost:3000` which nothing real could ever hit; original allowlist code kept commented (`#pragma warning disable/restore S125` around it) for an easy revert once a real frontend origin exists.

**Real bug found and never resolved**: Amazon SSM Agent never registered on either EC2 instance in this VPC (`aws ssm describe-instance-information` stayed empty for 20+ minutes; IAM role/route table/NACL/DNS-support/DHCP-options all confirmed correct by direct comparison against the working default-VPC case) — root cause not found. `rpm -qa | grep ssm` showed no SSM package installed on the AMI at all (`al2023-ami-*-x86_64`, resolved via `most_recent = true` at whatever moment each instance launched) — worth checking in a future session whether this specific AMI variant genuinely lacks the agent, or something else is wrong. **Deploy had to go through a temporary SSH key** (`rentifyx-deploy-temp`, imported via `aws ec2 import-key-pair`, `ssh_key_name` var wired through both app repos' `modules/ec2`) instead of the intended SSM-based flow — this key pair and the SG's SSH ingress rule are torn down along with everything else at the end of this session; if a future session needs interactive EC2 access again, this SSM gap needs a real fix (or re-import a fresh key) rather than assuming SSM will "just work."

## 2026-07-21 session: end-to-end flow confirmed working, kafka-ui replaces the broken topic-provisioning path, four real bugs fixed and verified live

**All 4 previously-identified issues from the 2026-07-20 retrospective were fixed and confirmed working on a fresh apply** (see `rentifyx-identity-api`/`rentifyx-communications-api` STATE.md for the code-level detail): `ConnectionStrings__kafka` now wired through `userdata.sh.tpl`/`deploy.yml` (resolved at runtime from this repo's SSM parameter, never hardcoded); `module.kafka_topics` removed entirely (see below); `deploy.yml`'s `EC2_INSTANCE_ID` now resolved dynamically; the MSK `MSKGroup`/`MSKTopicReadWrite` IAM resource ARNs now include the cluster-UUID wildcard segment (`.../<cluster>/*/*`) matching AWS's documented shape exactly — **this fix worked**: the consumer group went from erroring with `Access Denied` on `DescribeGroup`/`AlterGroup` to `STATE: STABLE` with partitions assigned.

**`module.kafka_topics` deleted for good** (was never going to work - see 2026-07-20 entry above for the SASL incompatibility). Replaced entirely by `kafka-ui` (this repo's `scripts/start-kafka-ui.sh` + `docs/kafka-ui.md`, added 2026-07-20 as a separate feature): kafbat-ui's built-in `aws-msk-iam-auth` (the official Java library) authenticated successfully on the very first attempt against the same cluster the Go-based Terraform provider couldn't handle - "Successfully logged in" in its own logs, no crash, no auth error. All 6 topics were created via kafka-ui's REST API (`POST /api/clusters/rentifyx/topics`) in one command each - far simpler than the throwaway C# tool from the previous session. **This is now the recommended path for any future topic provisioning** until/unless the Terraform provider issue gets root-caused.

**A second, previously-undiscovered real bug was found and fixed during this session's live testing**: `NotificationRequestedConsumer`, `RetryTopicConsumer` (×3, one per retry topic), and `DlqObserverHostedService` in `rentifyx-communications-api` all shared one literal `ConsumerGroupId` from config while each subscribed to a different single topic. This is what was actually causing messages to sit unconsumed even after the IAM ARN fix - Kafka's range assignor, given heterogeneous subscriptions inside one shared group, left `notification-requested`'s partitions technically "assigned" (group showed `STABLE`) but never delivered to anything reading them. Fixed by giving each consumer its own derived group ID (`{ConsumerGroupId}-main`, `-{retryTopicName}`, `-dlq`) - confirmed live: the main consumer picked up and processed a message that had been stuck for 12+ minutes, within seconds of redeploying.

**End-to-end flow confirmed working for real, for the first time across both sessions**: `rentifyx-identity-api` register/forgot-password → Kafka (`notification-requested` topic) → `rentifyx-communications-api` consumes → SES send → `Status: Sent` in DynamoDB → **real email delivered and confirmed rendering correctly** in the recipient's inbox. Two more real bugs surfaced and were fixed along the way (see `rentifyx-communications-api` STATE.md): the SES account is in **sandbox mode** (`ProductionAccessEnabled: false` - a production-access request was submitted via `aws sesv2 put-account-details` and was auto-denied instantly, case ID `178464532000461`; a real AWS Support Center case with more justification is the next step, not something the CLI can push further) - every test recipient needs its own SES identity verification until that's resolved; and `SesEmailSender` was sending the rendered HTML template as `Body.Text` instead of `Body.Html`, so recipients saw raw HTML source instead of a formatted email.

**All AWS infrastructure across all three repos was destroyed again at the end of this session** (same `terraform destroy` pattern as 2026-07-20 - comms-api, then identity-api, then platform). Next time infra goes up: the 4 fixes above should make the deploy clean on the first try (they did this session), topics can be created via kafka-ui in under a minute, and the only remaining known gaps are the unresolved SSM Agent registration issue (still needs the SSH-key workaround) and the SES sandbox limitation (needs either per-recipient verification for testing, or a proper AWS Support Center production-access request).

**Result: identity-api → Kafka → comms-api was not confirmed working end-to-end.** Topics existed, producer connected, but two more real bugs surfaced and were still open when the session ended: (1) `rentifyx-identity-api`'s `OutboxPublisher` stopped attempting new entries entirely after its first entry permanently failed (before topics existed) — a second, fresh registration produced zero log output at all, suggesting the background polling loop may have silently died rather than just failing to retry; (2) `rentifyx-communications-api`'s Kafka consumer got `SASL authentication error: Access denied` specifically from the **GroupCoordinator** (distinct from the topic-level auth that had been working) — the IAM policy JSON looked correct on inspection but something in the real authorization decision disagreed. **All AWS infrastructure for this session was destroyed afterward** (see `terraform destroy` note below) — these two bugs need to be reproduced fresh next time real infra exists to debug against.
